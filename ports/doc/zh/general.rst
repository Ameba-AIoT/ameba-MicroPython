Ameba port 通用信息
=====================

Ameba RTOS 是 Realtek（瑞昱）旗下一系列自带 WiFi（大部分型号还带 Bluetooth LE）的
System-on-Chip（SoC）产品线。本 MicroPython port 面向 Ameba RTOS SDK 统一的构建系统
（``ameba.py``）；某一份固件跑在家族里哪颗芯片上，是编译期的选择（见下文）。

Ameba 家族里的多颗芯片
------------------------

本 port 可跨 Ameba 家族里的多颗芯片运行。各芯片的能力和可用引脚有所不同，所以
一份固件镜像总是针对某一颗特定芯片和某一块特定板卡编译的。

编译目标是通过 ``ports/ameba-rtos-m/soc_info.json`` 里的 ``soc.name`` 字段
选择的；这是一个编译期选择，不是运行期可以切换的。

正因如此，下面的具体数字（引脚数量、外设数量、默认引脚分配）描述的都是这份固件
实际编译面向的那颗芯片和那块板卡；换一颗芯片可能会不一样。你那块板卡上印着的
名字永远是最权威的参照。

技术规格与 SoC 规格资料
-------------------------

Ameba 家族各颗芯片和模组的规格资料——架构、主频、内存、外设数量、封装/引脚
图——查 Realtek 官方的规格资料索引：
https://aiot.realmcu.com/zh/datasheet/index.html
这些是芯片能力和引脚定义的权威参考；至于板级细节（哪些引脚实际引出、模组
摆放位置等），也请准备好你那块板卡自己的规格资料和电路图。

下面这些是 *这份 MicroPython port* 实际暴露给 Python 的能力——不是芯片全部
的能力：

* GPIO 编号用的是 SDK 的 ``PinName`` 方案，不是从零开始的 GPIO 序号——具体
  看下面的 `引脚命名与编号`_。
* 本 port 暴露的外设（数量与默认值反映编译目标——见
  `Ameba 家族里的多颗芯片`_）：2 路 UART、2 路硬件 I2C（每路也都能切
  到 target/从机模式）、2 路硬件 SPI、一组 8 通道 PWM（所有通道共用同一个
  频率）、一组 8 通道 ADC、2 路 I2S（SPORT0/SPORT1）、一个硬件定时器池，以及
  一个硬件看门狗。

引脚命名与编号
---------------

本 port 里 ``machine.Pin`` 的编号 **不是** 简单的从零开始的 GPIO 序号。内部每个
引脚由 SDK 的 ``PinName`` 枚举标识，A 组引脚占用 0-31（``PA_0`` = 0、``PA_1`` =
1、... ``PA_31`` = 31），B 组引脚占用 32-63（``PB_0`` = 32、... ``PB_31`` = 63）。
把这个原始整数传给 ``Pin()`` 是能用的，但这意味着比如 ``Pin(6)`` 指的是
``PA_6``，不是 "GPIO6"，而且 ``PA_0``-``PA_5`` （索引 0-5）里的大部分会直接被
拒绝——这些是 SoC 的 SPI flash 控制器（SPIC）用来接模组内置那颗 SPI NOR flash
的引脚，永远不能当通用 GPIO 用。

因此，请始终优先用下面两种带名字的写法，而不是裸整数：

* ``Pin("PA14")`` —— 把引脚名当字符串传进去。这个名字直接对应板卡排针上印的
  丝印——排针上标着 ``PA16`` 的那个孔，代码里就是 ``Pin("PA16")``，不需要
  另外查表翻译。
* ``Pin(Pin.board.PA14)`` —— 同样一套名字，用 ``Pin.board`` 的属性形式而不是
  字符串。

这两种写法都是按板卡区分的，不只是按芯片区分：能接受的名字集合取决于这份固件
是给哪块板卡编译的，换一块板卡可能是不同的一套。请用你实际在跑的这块板卡上
印着的名字。

传入范围外或者其他非法的引脚（包括 ``PA_0``-``PA_5`` 里的任何一个）会抛出
``ValueError``，不会静默不生效。

软件外设 vs 硬件外设
----------------------

有几条总线同时提供硬件加速版本和几乎能在任意 GPIO 上跑的软件（bit-bang）版本：
:class:`machine.SPI` / :class:`machine.SoftSPI`，以及 :class:`machine.I2C` /
:class:`machine.SoftI2C`。当你的引脚正好落在外设的固定或默认布线上时，用硬件版本；
需要任意引脚，或者硬件总线的两个实例不够用时，用 ``Soft*`` 版本。

底层框架
---------

本 port 上的 MicroPython 是作为一个任务跑在
`ameba-rtos SDK <https://github.com/Ameba-AIoT/ameba-rtos>`_ 之上的，这是
Realtek 面向 Ameba 家族的、基于 FreeRTOS 的开发框架。外设驱动、网络协议栈，
以及本 port 依赖的构建系统（``ameba.py``），都由这个 SDK 提供。
